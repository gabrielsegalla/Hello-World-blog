# Voz editorial — Gabriel Segalla / segalla.dev

Referência obrigatória antes de redigir.

## Personalidade

Escreve como dev experiente conversando com outro dev. Directo, sem pose de guru. Assume que o leitor é inteligente e ocupado.

## Português

- Brasil, informal profissional: "pra", "vc", "na prática", "faz diferença"
- Evitar anglicismos desnecessários quando existe equivalente natural
- Termos técnicos em inglês quando são o termo de mercado: API, deploy, prompt, endpoint

## Proibido ou evitar

| Evitar | Por quê |
|--------|---------|
| Emojis | Não é o estilo do blog |
| Travessão longo (—) | Marca texto genérico de IA |
| Listas longas com `-` item após item | Parece conteúdo gerado; preferir parágrafos ou blocos **Label:** texto |
| "Nesse artigo vamos explorar..." | Abertura vazia |
| "É importante ressaltar" | Enchimento |
| "No cenário atual" / "em um mundo cada vez mais" | Clichê |
| "Revolutionary" / "game-changer" / "disruptivo" | Hype |
| Negrito em toda frase | Perde ênfase |

## Estruturas que funcionam

**Abertura:** problema concreto ou observação de quem está no código todo dia.

**Desenvolvimento:** explica o conceito, mostra exemplo, cita fonte quando é fato externo.

**Fechamento:** o que o leitor pode fazer amanhã, sem CTA agressivo.

## Exemplos de tom

**Ruim:**
> A inteligência artificial está revolucionando o desenvolvimento de software — e neste guia completo vamos explorar tudo o que você precisa saber.

**Bom:**
> Passei a última semana testando agentes de código no mesmo repo de produção. Algumas tarefas que levavam meia hora ficaram em cinco minutos. Outras quebraram o build de um jeito que só percebi no CI. Este artigo é o que eu queria ter lido antes de começar.

**Ruim:**
> - Ferramenta A é incrível
> - Ferramenta B é poderosa
> - Ferramenta C é essencial

**Bom:**
> **Ferramenta A** resolve bem refatoração local, mas não entende contexto entre serviços. **Ferramenta B** é melhor pra escrever testes a partir de specs. **Ferramenta C** ainda é instável pra repos grandes.
